## 项目概述

目前我校跳蚤市场建立在 QQ 群上。在群里，卖家通常都是将商品拍照后附价格，而附文字描述较少，对于二手书尤甚。卖方这样很方便，但买方就比较麻烦，需要一张张图片找而不能搜索，而且很可能找到别人已经售出的商品。

因此我想开发这样一个跳蚤市场应用：卖方可以扫描书本条形码并填写售价、书况等信息，也可以对一本或多本书封面或书脊拍照后调用 LLM 批量添加，然后将信息上传到服务器数据库；对其他商品也同样可以提供拍照识别等功能，填写信息后上传到服务器。买方则可以搜索和筛选自己需要的商品，并获取卖方联系方式。相应地，买方也可以发布“收购”需求，等待卖方联系。

## 设计目标

数据库系统负责保存用户、登录会话、出售/求购发布、书籍信息、图片元数据、标签、通知和 ISBN 查询缓存，以用于账户管理、发布管理、条件检索、站内通知等业务功能。

数据库系统应达到以下目标：

1. 将分散在聊天消息中的商品和求购信息转化为结构化、可检索的数据。
2. 统一管理出售和求购两类发布，同时支持普通商品与书籍的差异化信息。
3. 保证用户、发布、标签、图片、书籍和通知之间关系的一致性。
4. 支持按时间浏览、关键字搜索、标签筛选、价格筛选和交易状态筛选。
5. 支持发布者维护自己的发布，并在并发修改数量时避免覆盖更新。
6. 保护学号、密码、会话令牌和联系方式等敏感信息。
7. 缓存 ISBN 查询结果，减少对外部书籍信息接口的重复调用。

## 由数据库管理的数据

- 校内用户账号、角色、状态及注销标记。
- 登录会话、失效时间和撤销状态。
- 出售与求购发布及其交易数量、价格、联系方式、有效期和状态。
- 书籍类发布的 ISBN、书名、作者、出版社和版本信息。
- 发布图片的存储键、显示顺序和尺寸元数据。
- 标签及发布与标签之间的多对多关系。
- 个人通知、全员公告的数据表达、未读状态和分页读取。
- 外部 ISBN 查询结果缓存。
- 为检索、分页、软删除和业务查询服务的索引、触发器与事务。

## 参与者与外部系统

| 参与者 | 职责与权限 |
| --- | --- |
| 未登录访客 | 浏览出售/求购摘要，进行搜索、筛选和分页；不能查看包含联系方式等敏感信息的完整详情 |
| 已认证学生用户 | 注册、登录、查看详情、发布出售/求购、维护本人发布、查看通知、修改账号信息和注销账号 |
| 发布所有者 | 编辑或软删除本人发布，调整已售/已收数量，查看包含过期和售罄项在内的本人发布 |
| 系统定时任务 | 扫描即将过期的发布，为所有者创建临期提醒 |
| BIT 101 认证服务 | 在注册或重置密码前验证校内身份 |
| ShowAPI 书籍服务 | 在本地 ISBN 缓存未命中时提供书籍元数据 |

## 业务流程

### 注册、登录与会话

1. 用户通过学校的统一身份验证完成校内身份验证后，才能注册 BitMart 账号或重置密码。
2. 同一学号在未注销状态下只能对应一个有效账号。
3. 密码只保存哈希，不得保存明文。
4. 登录成功后签发不透明会话令牌；服务端只保存令牌的哈希值、所属用户、创建时间、最近使用时间、到期时间和撤销状态。
5. 会话默认有效期为 30 天。过期、被撤销或所属用户已注销的会话不得通过认证。
6. 重置密码和注销账号时，应撤销该用户的全部有效会话。
7. 注销采用软删除，即注销后历史数据仍物理保留，但标记为已删除并不再暴露给用户；同一学号可重新注册。

### 发布出售与求购

1. 已登录用户可以发布 `SELL`（出售）或 `BUY`（求购）信息，两者使用相同的核心数据结构。
2. 发布分为 `GENERAL`（普通商品）和 `BOOK`（书籍）两种类别。
3. 每条发布应记录标题、描述、总数量、已售/已收数量、单价、可选原价、联系方式、取货地点、标签、图片、有效期、创建时间和更新时间。
4. 单价允许为空，表示面议或带价联系；原价允许为空，表示不展示划线价。
5. 书籍类发布可以附加 ISBN、书名、作者、出版社和版本信息。
6. 图片先上传到文件存储，数据库保存返回的文件键及顺序。客户端单条发布最多选择 9 张图片并在上传前压缩；服务端上传接口默认限制单个文件不超过 5 MiB。
7. 用户可以一次提交多条草稿。批量发布必须具有原子性：全部成功或全部回滚。

### 浏览、搜索与筛选

1. 系统分别提供出售和求购列表，并按创建时间与编号倒序返回。
2. 列表采用游标分页，客户端将服务端返回的游标原样用于下一页查询。
3. 公开列表默认排除已软删除、已过期和已售罄/已收满的发布。
4. 用户可以按关键字、标签、价格区间、是否包含无定价发布、是否包含售罄发布进行筛选。
5. 关键字搜索范围至少包括标题和描述，并兼顾中文全文检索与模糊匹配。
6. 多个标签的筛选语义为“命中任一所选标签”。
7. 未登录访客只能获得发布摘要；联系方式、完整描述、取货地点及书籍详情等完整信息仅向已登录用户返回。

### 本人发布管理

1. 用户可以查看自己的全部未删除发布；默认允许包含已过期和已售罄/已收满项目。
2. 用户可以编辑标题、描述、价格、原价、总数量、取货地点、联系方式、有效期、标签、图片、类别和书籍信息。
3. 用户可以增加或减少已售/已收数量，但其值必须位于 0 与总数量之间。
4. 当总数量和已售/已收数量同时修改时，最终结果仍必须满足数量约束。
5. 已售/已收数量更新必须原子执行；重叠事务内读到的数量已被其他事务修改时，条件更新应报告冲突并回滚同一事务中的其他修改。当前 API 未携带客户端版本号，因此不保证识别前一事务已经提交后才到达的陈旧客户端请求。
6. 删除发布采用软删除，不立即移除历史记录及其关联元数据。

### ISBN 查询与书籍信息

1. Android 端扫描 EAN-13 条形码，提取 ISBN 码发送给服务端查询书籍相关信息；查询失败时只好由用户手工填写。
2. 服务端应优先查询本地 `book_meta` 缓存；命中时直接返回。
3. 缓存未命中时，服务端调用 ShowAPI；成功后保存原始响应和常用书籍字段。
4. 外部接口不可用时，不影响注册、登录、普通发布、列表和通知等核心功能；用户仍可手工填写书籍信息。
5. 发布中的书籍信息是发布时的业务快照，不要求与缓存表建立强制外键关系。

### 通知

1. 系统应统一表达个人通知和全员公告；个人通知关联具体用户，全员公告的用户关联为空。
2. 当前版本自动产生的主要通知为发布临期提醒。
3. 用户可以分页拉取通知、查询个人未读数并将个人通知标记为已读。
4. 全员公告不计入个人未读数，也不记录逐用户已读状态。
5. 通知负载可以保存关联发布编号、到期时间等结构化跳转信息。

## 数据需求

### 服务端数据对象

| 数据对象 | 主要数据 | 生命周期 |
| --- | --- | --- |
| 用户 | 学号、密码哈希、昵称、角色、状态、创建/注销时间 | 注册时创建，注销时软删除 |
| 会话 | 令牌哈希、用户、创建/使用/过期时间、终端信息、撤销状态 | 登录时创建，退出/改密/注销时撤销，过期后不可用 |
| 发布 | 类型、类别、标题、描述、价格、数量、地点、联系方式、有效期、来源和时间戳 | 发布时创建，编辑时更新，删除时软删除 |
| 发布图片 | 发布、文件键、顺序、宽高 | 随发布创建或编辑替换；物理删除发布时可级联 |
| 标签 | 归一化名称、使用次数 | 首次使用时创建，可被多条发布复用 |
| 发布标签关联 | 发布、标签 | 随发布的标签变化而增删 |
| 发布书籍信息 | 发布、ISBN、书名、作者、出版社、版本 | 仅书籍类发布使用，是发布时快照 |
| ISBN 缓存 | ISBN、常用字段、原始响应、抓取时间 | 查询成功时写入，后续查询复用 |
| 通知 | 接收用户、类别、标题、正文、负载、已读/创建时间 | 由系统任务或公告业务创建，标记已读后保留 |
| 推送令牌（预留） | 用户、令牌、平台、创建/更新时间 | 现有物理结构中的未来扩展，不属于本期业务实现及验收对象 |

### 数据关系

- 一个用户可以拥有多个会话、发布和个人通知；预留结构以允许未来为用户绑定多个推送令牌。
- 一条发布只属于一个用户；一个用户可以发布多条出售或求购信息。
- 一条发布可以有多张图片，图片在同一发布内具有唯一顺序。
- 发布与标签为多对多关系。
- 一条发布最多对应一条书籍扩展记录。
- ISBN 缓存可被多条书籍发布在业务上复用，但发布书籍信息保存独立快照。
- 全员公告不属于某个具体用户。

## 业务规则与数据约束

| 编号 | 规则 |
| --- | --- |
| BR-01 | 活跃账号的学号必须唯一；已注销账号不阻止同学号重新注册 |
| BR-02 | 昵称可空，空昵称在界面上显示为“匿名”；非空昵称最长 32 个字符 |
| BR-03 | 发布标题不能为空且最长 32 个字符；描述最长 1024 个字符 |
| BR-04 | 总数量范围为 1～9999；已售/已收数量范围为 0～总数量 |
| BR-05 | 单价和原价可空；非空时范围为 0～99999999.99 |
| BR-06 | 每条发布至少包含一个非空联系方式 |
| BR-07 | 每条发布最多 8 个标签；标签去首尾空白、转小写并压缩连续空白后去重；标签名最长 20 个字符 |
| BR-08 | 发布默认有效期为 30 天，可配置范围为 1～365 天；绝对日期必须晚于当前时间且不超过最大期限 |
| BR-09 | 公开查询不返回已删除、已过期项目；是否包含售罄项目由筛选条件决定 |
| BR-10 | 只有发布所有者或具备服务端管理权限的角色可以修改、删除发布 |
| BR-11 | 个人通知只能由所属用户标记已读；公告不参与个人已读计数 |
| BR-12 | 书籍扩展记录与发布是一对零或一关系；普通商品不应保留书籍扩展数据 |
| BR-13 | 图片顺序在同一发布内唯一，发布与标签的组合唯一 |
| BR-14 | 所有时间统一保存为带时区时间，避免客户端时区差异造成过期判断不一致 |

## 非功能需求

### 一致性与可靠性

- 发布、书籍扩展、图片和标签关联的创建应处于同一事务。
- 批量发布应在单一事务中完成。
- 修改发布涉及主表、标签、图片和书籍扩展时，应全部提交或全部回滚。
- 重置密码和注销账号涉及的账号、会话和发布状态变化应保持原子性。
- 数据定义由版本化迁移脚本统一管理，应用 ORM 映射不得成为独立的结构来源。

### 性能

- 常用公开列表、本人发布、会话校验、通知分页和标签筛选应有适配索引。
- 列表和通知采用 keyset 游标分页，避免深页 `OFFSET` 带来的性能衰减。
- 标题和描述应支持 GIN 全文检索与三元组模糊匹配。
- ISBN 缓存应避免同一 ISBN 的频繁外部请求。
- 数据库连接由连接池管理，默认最大连接数为 10。

### 安全与隐私

- 密码采用 Argon2id 哈希；服务端会话只保存随机令牌的 SHA-256 哈希。
- 未登录访问不得获得联系方式等完整详情。
- Android 端 LLM API Key 和常用联系方式不得写入 BitMart 服务端数据库。
- 数据库账号应遵循最小权限原则，生产环境不得继续使用示例口令。
- JSON 数据写入前必须经过应用层结构与非空校验。

### 可维护性与可扩展性

- 使用 PostgreSQL 16 和 Flyway 管理数据库版本。
- `type`、`category`、`role`、`status`、`source` 等编码在应用层应集中定义，避免不同模块解释不一致。
- 通知负载、联系方式和外部原始响应使用 JSONB，以便在不频繁修改表结构的情况下承载半结构化数据。
- 数据结构允许未来增加推送、管理员操作和新的发布来源，但预留结构不等同于当前已实现功能。
