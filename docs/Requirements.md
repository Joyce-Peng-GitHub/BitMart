## 项目概述

目前我校跳蚤市场建立在 QQ 群上。在群里，卖家通常都是将商品拍照后附价格，而附文字描述较少，对于二手书尤甚。卖方这样很方便，但买方就比较麻烦，需要一张张图片找而不能搜索，而且很可能找到别人已经售出的商品。

因此我想开发这样一个跳蚤市场应用：卖方可以扫描书本条形码并填写售价、书况等信息，也可以对一本或多本书封面或书脊拍照后调用 LLM 批量添加，然后将信息上传到服务器数据库；对其他商品也同样可以提供拍照识别等功能，填写信息后上传到服务器。买方则可以搜索和筛选自己需要的商品，并获取卖方联系方式。相应地，买方也可以发布“收购”需求，等待卖方联系。

## 需求目标

数据库系统负责保存账号信息、登录状态、出售/求购发布、书籍信息、图片信息、标签、通知和书籍查询结果，为账号管理、发布管理、条件检索和站内通知等业务提供数据支持。

数据库期望达到以下目标：

1. 统一管理出售和求购两类发布，同时支持普通商品与书籍的差异化信息。
2. 保证用户、发布、标签、图片、书籍和通知之间的业务关系一致。
3. 支持按发布时间浏览，以及按关键字、标签、价格和交易状态筛选。
4. 支持发布者维护自己的发布，并避免并发操作产生不一致结果。
5. 保护学号、认证凭据和联系方式等敏感信息。
6. 复用已经取得的书籍信息查询结果，减少无意义的重复调用 API。

## 需要持久化的业务信息

- 校内用户账号、显示信息、角色、状态和注销状态。
- 用户登录状态及其有效性。
- 出售/求购发布的商品/需求信息。
- 发布所关联的图片及其展示顺序。
- 标签及发布与标签之间的关联。
- 个人通知、全员公告及个人通知的已读状态。
- 外部书籍服务查询结果。

## 角色与权限

项目中，角色与权限实际由业务层代码管控。

| 角色 | 权限 |
| --- | --- |
| 未登录访客 | 浏览出售/求购摘要，进行搜索、筛选和分页；不能查看包含联系方式等敏感信息的完整详情 |
| 已认证学生用户 | 注册、登录、查看详情、发布出售/求购、维护本人发布、查看通知、修改账号信息和注销账号 |
| 发布所有者 | 编辑或删除本人发布，调整已售/已收数量，查看包含过期和售罄项在内的本人发布 |
| 系统定时任务 | 扫描即将过期的发布，为所有者创建临期提醒 |

## 外部服务

- BIT 101 认证服务：由 BIT 101 服务器转接北理工的统一身份验证系统验证校内身份。
- ShowAPI 书籍服务：根据 ISBN 查询书籍元数据。

## 业务流程

### 注册、登录与会话

1. 用户通过学校的统一身份验证完成校内身份验证后，才能注册 BitMart 账号或重置密码。
2. 同一学号在未注销状态下只能对应一个有效账号。
3. 系统必须安全保存用户的认证凭据，不得保存或泄露可直接用于登录的明文密码。
4. 登录成功后，系统应建立具有有效期且可以主动失效的登录状态。
5. 登录状态过期、被撤销或所属用户已注销后，不得继续通过认证。
6. 重置密码和注销账号时，应使该用户已有的登录状态全部失效。
7. 用户注销后不再作为有效用户参与业务，其历史数据应予保留；同一学号可以重新注册。

### 发布出售与求购

1. 已登录用户可以发布出售或求购信息。
2. 发布分为普通商品和书籍两种类别。
3. 每条发布应记录标题、描述、总数量、已售/已收数量、单价、可选原价、联系方式、取货地点、标签、图片、有效期、创建时间和更新时间。
4. 价格允许为空，表示面议或带价联系；原价也允许为空，因为不重要或忘记了。
5. 书籍类发布可以附加 ISBN、书名、作者、出版社和版本信息。
6. 一条发布可以包含多张图片，并保持用户设定的展示顺序。
7. 用户可以一次提交多条草稿；批量发布时应全部成功或全部失败。

### 浏览、搜索与筛选

1. 系统分别提供出售和求购列表，并按发布时间倒序展示。
2. 列表支持分页浏览。
3. 公开列表默认不展示已删除、已过期和已售罄/已收满的发布。
4. 用户可以按关键字、标签、价格区间、是否包含无定价发布、是否包含售罄发布进行筛选。
5. 关键字搜索范围至少包括标题和描述。
6. 系统应支持同时选择多个标签进行筛选。
7. 未登录访客只能获得发布摘要；联系方式、完整描述、取货地点及书籍详情等完整信息仅向已登录用户返回。

### 本人发布管理

1. 用户可以查看自己的全部未删除发布；默认允许包含已过期和已售罄/已收满项目。
2. 用户可以编辑标题、描述、价格、原价、总数量、取货地点、联系方式、有效期、标签、图片、类别和书籍信息。
3. 用户可以增加或减少已售/已收数量，系统应拒绝不符合当前总数量的修改。
4. 并发修改交易数量时，系统应避免产生超出总数量或部分更新的结果。
5. 删除后的发布不再对用户展示，但应保留必要的历史数据。

### ISBN 查询与书籍信息

1. 用户可以扫描书籍条形码提取或手工输入 ISBN 来查询书籍信息。
2. 系统应复用已经取得的书籍查询结果，避免不必要的重复外部查询。
3. 查询成功时，书籍信息可以用于填写发布内容，用户仍可根据实际情况修改。
4. 查询失败或外部服务不可用时，用户可以手工填写书籍信息，其他核心功能不受影响。

### 通知

1. 系统应统一管理面向具体用户的个人通知和面向所有用户的公告。
2. 当前版本自动产生的主要通知为发布临期提醒。
3. 用户可以分页拉取通知、查询个人未读数并将个人通知标记为已读。
4. 全员公告不计入个人未读数，也不记录逐用户已读状态。
5. 临期提醒应能够关联到对应发布，以便用户查看和处理。

## 数据需求

| 数据对象 | 业务用途 |
| --- | --- |
| 用户 | 标识校内用户，保存个人显示信息和账号状态 |
| 登录状态 | 判断用户是否已经通过认证以及是否仍可访问受保护功能 |
| 发布 | 表达出售或求购内容及其当前交易进度 |
| 发布图片 | 为发布提供图片展示 |
| 标签 | 对发布进行分类、筛选和热门标签统计 |
| 发布书籍信息 | 保存书籍类发布所需的专有信息 |
| 书籍查询结果 | 复用从外部服务取得的书籍元数据 |
| 通知 | 保存个人提醒、公告和个人已读状态 |

## 非功能需求

### 一致性与可靠性

- 一次业务操作涉及的相关数据应保持一致，不得留下部分完成的状态。
- 批量发布应全部成功或全部失败。
- 重置密码、注销账号和编辑发布等操作失败时，不得遗留相互矛盾的数据。
- 系统异常或外部书籍服务不可用时，既有核心数据仍应可正常使用。

### 性能

- 各操作应在可接受的时间内完成。
- 关键字搜索需要满足中英文标题和描述的实际检索需要。
- 重复查询同一 ISBN 时，应尽量避免重复访问外部服务。

### 安全与隐私

- 用户密码和登录凭据必须用哈希保存。
- 未登录访问不得获得联系方式等完整详情。
- Android 端 LLM API Key 和常用联系方式不得写入 BitMart 服务端数据库。
- 普通用户只能增删改自己的发布。
- 注销后的用户和失效的登录状态不得继续访问受保护功能。

### 可维护性与可扩展性

- 数据结构应能够随项目版本演进，并保证升级过程可追踪、可重复。
- 不同模块对用户、发布、通知等业务状态的含义应保持一致。
- 系统应允许未来扩展推送、管理员操作和新的发布来源，而不影响当前核心业务。
- 尚未实现的扩展能力不得被当作当前功能对外提供。
