我的 Android 开发课程需要在四周时间内完成一个结课项目。目前我的想法如下。

我注意到学校的跳蚤市场 QQ 群里，卖家通常都是将商品拍照后附价格，而很少附文字描述，对于二手书尤甚。卖方这样很方便，但买方就比较麻烦，需要一张张图片找，而且可能找到别人已经售出的商品。因此我想开发这样一个跳蚤市场应用：卖方可以扫描书本条形码并填写售价（允许留空，即买方带价联系）、书况等信息，也可以对一本或多本书封面或书脊拍照后调用 OCR 或 LLM 批量添加，然后将信息上传到服务器数据库。对其他商品也最好提供拍照识别等功能，填写信息后上传到服务器。买方则可以搜索自己需要的商品（兼顾商品名、商品描述等字段（有可能中英文混合），最好能模糊搜索，允许设置期望价格等），并获取卖方联系方式。商品可以添加 tag 并允许通过标签筛选。同一条出售单可能包含多件同类商品并且逐个售卖。

进一步开发方向（如果有时间）：
- 可以添加买方的求购功能，并在有人恰好挂卖相关商品时发送应用内通知甚至手机通知。
- 最新动态（新上市、新需求）。
- 好物推荐（如果可行）。
- 如果得到管理员授权，可以通过 NapCat 获取现有跳蚤市场群信息，通过 LLM 提取要素后提交到服务器。

我们的平台必须绝对避免交易，即要求用户自行进行交易，以避免超出我们能力范围的法律责任。

为了避免社会人员混入，要求用户在注册账号时必须提供北京理工大学统一身份认证系统的账号密码（完成验证才注册账号），我们可以调用BIT101 项目提供的认证接口（[文档]( https://bit101-api.apifox.cn/)）进行身份验证；后续登录时提供学号和平台密码（用哈希值存储）即可。为了与现有 QQ 群兼容，应特别允许来自 NapCat 方式的商品只关联 QQ 号而不关联我们的平台账号，向用户提示风险即可。

OCR 功能最好采用本地组件。LLM API 采用 OpenAI Compatible 协议。允许用户调用自己的 LLM API（保存在客户端本地，客户端直接向 LLM API 发请求）或上传到服务端调用我们的 API。允许用户在 OCR 和 LLM 选择。

课程要求 Android 端采用 Jetpack Compose；应当采用现代 UI 风格，需要支持中英双语。后端优先也用 Kotlin。

我拥有自己的内网 Linux 服务器。我朋友有带公网 IP 和域名的 Linux 服务器，且我的 Tailscale 跟他的服务器组了网络。可以让他将转发请求到我的服务器，也可以直接将服务部署在他的服务器上。

我将项目命名为 BitMart。考虑到我们的规模不大，可以采用相对轻量的方案，但需要容易改进到高性能方案。
